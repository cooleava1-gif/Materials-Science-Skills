# materials-paper-to-patent

**Version:** 1.0.0

Convert materials science papers into evidence-grounded Chinese invention
patent drafts. Default invention type is **process-material**; switch via
`manifest.yaml` axes for algorithm, apparatus, or mixed inventions.

把材料科学研究论文转为**证据可溯源**的中国发明专利申请草稿。默认
`invention_type` 是 **process-material**（配方/工艺/材料类发明专利），可
通过 manifest axes 切换为 algorithm / apparatus / mixed。

**Built from** — A 3-axis routing system, a curated civil-patent knowledge
base, a 7-rule claims-content validator, 9 Python scripts, 5 references, 1
worked example, and 3 unit tests.

## Three-axis routing

| Axis | Values | Default |
|---|---|---|
| `source_format` | `pdf-text` / `scanned-pdf` / `pasted-text` / `mixed-project` | `pdf-text` |
| `task_mode` | `full-draft` / `claim-set` / `disclosure-analysis` / `paper-patent-audit` | `full-draft` |
| `invention_type` | `process-material` / `apparatus-system` / `algorithm-software` / `mixed` | `process-material` |

The default `invention_type = process-material` (rather than the more
common `algorithm-software` in the nature-skills upstream) reflects the
civil-engineering and construction-materials focus of this bundle.

## Civil-patent knowledge base — `static/core/patent_kb.yaml`

The skill ships a curated KB that the claims-content validator reads:

| KB section | Content |
|---|---|
| Patent law articles | Chinese Patent Law articles 22, 26.3, 26.4, 31.1, 33 + Implementing Regulations article 20 |
| CNIPA examination guidelines | Examination guidelines 2023 edition, second-part substantive examination |
| Invention-type verb patterns | 4 patterns (process-material, apparatus, system, mixture-formula) |
| Claim anti-patterns | 9 anti-patterns (algorithm-software fallback, missing technical features, unsupported overclaims, …) |
| Unit-alias groups | 7 groups (strength / modulus / composition / temperature / time / size / size-micro) |
| Material-domain links | 6 links: civil_cement_concrete / civil_asphalt / civil_insulation / ceramics_structural / polymer_composite / metal_alloy |

## Claims-validation engine — `scripts/validate_patent_claims.py`

Runs 7 rule functions on every draft. Output severities:
`ERROR` (blocks submission) / `WARNING` (review before submit) / `INFO`
(advisory).

| Rule | Purpose |
|---|---|
| `check_independent_claim_technical_features` | Independent claims must carry concrete technical features |
| `check_dependent_claim_references` | Dependent claims must correctly reference their parent |
| `check_support_in_specification` | Each claim feature must be supported by the description |
| `check_anti_patterns` | No claim should fall into the 9 anti-pattern categories |
| `check_unit_consistency` | Units and aliases must be consistent across the document |
| `check_invention_type_alignment` | The verb pattern must match the declared `invention_type` |
| `check_claim_count_limits` | Total claim count stays within CNIPA-recommended limits |

`scripts/validate_patent_draft.py` is the structural companion (it checks
the `draft.json` schema, the section ordering, and the source-ID format).

## Pipeline

1. Detect `source_format`, `task_mode`, `invention_type` from the user
   request and `profile.yaml`.
2. Build source map, terminology ledger, evidence ledger.
3. Draft claims with claim-feature map (every claim feature must trace to
   a `source-id` of kind `P` (paragraph) / `E` (equation) / `F` (figure) /
   `C` (citation)).
4. Validate via `validate_patent_draft.py` (structure) +
   `validate_patent_claims.py` (content).
5. Build the DOCX package via `build_patent_package.py`.

|---|---|

| `render_flowchart_svg.py` | Main flowchart SVG (matplotlib) |
| `math_to_omml.py` | LaTeX → Office Math (OMML) |

## Outputs

`draft.json` (machine-readable structure), a complete DOCX application
(description + claims + abstract + cover letter), and a `flowchart.svg` for
the process diagram. Figure notes are produced as text; the actual figures
are generated by `materials-figure`.

## Worked example

`examples/civil-concrete-strengthening/` — a complete concrete-strengthening
patent package with `draft.json`, `flow-steps.json`, and `README.md`. Runs
end-to-end through all three validation gates (structural, claims content,
build package).

## Validation

- Unit tests (3):
  `plugins/materials-skills/skills/materials-paper-to-patent/tests/test_*.py`
- (A dedicated `evals/evals.json` and a `tests/pressure-tests/` directory
  are not yet shipped for this skill; the figure / reader / polishing /
  response skills each have their own `evals.json` that exercises the
  cross-skill flow.)
- Bundle verification:
  `python .\scripts\run_release_checks.py --json`

Run the validation engine against a draft:

```powershell
cd plugins/materials-skills/skills/materials-paper-to-patent
python scripts/validate_patent_claims.py examples/civil-concrete-strengthening/draft.json
```

Build the full DOCX package:

```powershell
python scripts/build_patent_package.py examples/civil-concrete-strengthening/draft.json
```
